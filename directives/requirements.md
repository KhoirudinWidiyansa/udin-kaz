# Requirements Document

## Introduction

This document specifies the requirements for enhancing the Kas Keluarga family finance application with six new features: spending trend visualization, category budgets, end-of-month spending prediction, browser notifications, split transactions, and planner optimization. These enhancements aim to provide better financial insights, proactive spending management, and improved user experience for family members tracking their expenses.

## Glossary

- **System**: The Kas Keluarga Next.js PWA application
- **User**: A family member using the application (Ayah, Ibu, Udin, etc.)
- **Transaction**: A financial record in the transaksi table (id, tanggal, jenis, nominal, kategori, nama, catatan)
- **Category**: A spending classification (kategori field in transaksi table)
- **Budget**: A spending limit set for a specific category for a given month
- **Split_Transaction**: A transaction divided into multiple categories with separate amounts
- **Trend_Chart**: A line chart visualization showing spending patterns over time
- **Notification**: A browser push notification sent to the User
- **Monthly_Target**: The total spending limit for a month
- **Projection**: A calculated estimate of total spending by end of month based on current patterns
- **Database**: PostgreSQL database containing transaksi and other tables
- **Insight_Tab**: The application tab displaying financial analysis and insights
- **Home_Tab**: The main application tab showing balance and transaction list
- **Planner**: The AI-powered spending scenario generator feature
- **Scenario**: A spending plan generated by the Planner (hemat, sedang, boros)

## Requirements

### Requirement 1: Spending Trend Visualization

**User Story:** As a User, I want to view spending trends over time in a line chart, so that I can understand my spending patterns and identify changes in behavior.

#### Acceptance Criteria

1. WHEN the User navigates to the Insight_Tab, THE System SHALL display a Trend_Chart showing spending over time
2. THE Trend_Chart SHALL support three view modes: daily, weekly, and monthly aggregation
3. WHEN the User selects a view mode, THE System SHALL update the Trend_Chart within 500ms
4. THE System SHALL provide filter controls for Category and family member name
5. WHEN the User applies a filter, THE System SHALL update the Trend_Chart to show only filtered data
6. THE Trend_Chart SHALL display nominal values on the Y-axis and time periods on the X-axis
7. THE Trend_Chart SHALL use responsive design to fit mobile and desktop screens
8. WHEN no transaction data exists for the selected filters, THE System SHALL display an empty state message

### Requirement 2: Category Budget Management

**User Story:** As a User, I want to set spending limits for each category, so that I can control expenses and receive warnings when approaching limits.

#### Acceptance Criteria

1. THE System SHALL create a new database table budget_kategori with columns: id, kategori, limit, bulan, created_at
2. THE System SHALL provide a user interface to set budget limits for each Category
3. WHEN a User sets a budget limit, THE System SHALL store the limit in the budget_kategori table with the current month
4. THE System SHALL calculate the percentage of budget used for each Category in real-time
5. WHEN budget usage is below 80%, THE System SHALL display a green indicator
6. WHEN budget usage is between 80% and 99%, THE System SHALL display a yellow indicator
7. WHEN budget usage is 100% or above, THE System SHALL display a red indicator
8. THE System SHALL display budget progress on both Home_Tab and Insight_Tab
9. THE System SHALL aggregate spending from the transaksi table grouped by Category and month
10. WHEN a new month begins, THE System SHALL reset budget tracking for the new month

### Requirement 3: End-of-Month Spending Prediction

**User Story:** As a User, I want to see a prediction of my total spending by month end, so that I can adjust my behavior if I'm projected to exceed my target.

#### Acceptance Criteria

1. THE System SHALL calculate daily average spending from the current month's transactions
2. THE System SHALL project total monthly spending using the formula: (daily_average × days_in_month)
3. THE System SHALL compare the Projection with the Monthly_Target
4. WHEN the Projection exceeds the Monthly_Target, THE System SHALL display a warning indicator
5. THE System SHALL display the Projection on the Home_Tab balance section
6. THE System SHALL update the Projection whenever a new Transaction is added
7. WHEN fewer than 3 days of data exist in the current month, THE System SHALL display a message indicating insufficient data
8. THE Projection SHALL use only pengeluaran transactions from the current month

### Requirement 4: Browser Push Notifications

**User Story:** As a User, I want to receive browser notifications for important spending events, so that I can stay informed without actively checking the app.

#### Acceptance Criteria

1. THE System SHALL request browser notification permission when the User first accesses the notification settings
2. THE System SHALL provide a settings interface to enable or disable notifications
3. WHEN a Category budget reaches 80% usage, THE System SHALL send a Notification
4. WHEN a Category budget reaches 90% usage, THE System SHALL send a Notification
5. WHEN a Category budget reaches 100% usage, THE System SHALL send a Notification
6. WHEN daily spending exceeds the daily average by 50%, THE System SHALL send a Notification
7. WHEN no Transaction has been logged today and the time is 8:00 PM, THE System SHALL send a reminder Notification
8. THE System SHALL store notification preferences in browser localStorage
9. THE Notification SHALL include the event type and relevant spending details
10. THE System SHALL use the PWA service worker to deliver notifications
11. WHEN the User clicks a Notification, THE System SHALL open the application to the relevant tab

### Requirement 5: Split Transaction Support

**User Story:** As a User, I want to split a single transaction across multiple categories, so that I can accurately categorize expenses that span multiple purposes.

#### Acceptance Criteria

1. THE System SHALL create a new database table transaksi_split with columns: id, transaksi_id, kategori, nominal, catatan, created_at
2. THE System SHALL add a "Split" button to the transaction form
3. WHEN the User clicks the "Split" button, THE System SHALL display an interface to add multiple category splits
4. THE System SHALL validate that the sum of split amounts equals the total transaction nominal
5. WHEN the User submits a Split_Transaction, THE System SHALL store the parent transaction in the transaksi table
6. WHEN the User submits a Split_Transaction, THE System SHALL store each split in the transaksi_split table with the parent transaksi_id
7. THE System SHALL display split details in the transaction list with an expandable view
8. THE System SHALL aggregate split amounts correctly in the category summary calculations
9. WHEN calculating budget usage, THE System SHALL include amounts from transaksi_split table grouped by Category
10. THE System SHALL allow the User to edit or delete Split_Transactions
11. WHEN a parent Transaction is deleted, THE System SHALL delete all associated transaksi_split records

### Requirement 6: Planner Optimization

**User Story:** As a User, I want the AI planner to show only 2 relevant scenarios instead of 3, so that I can make decisions faster without information overload.

#### Acceptance Criteria

1. THE System SHALL modify the /api/planner route to generate only 2 scenarios
2. THE System SHALL generate a "hemat" (economical) Scenario
3. THE System SHALL generate a "sedang" (moderate) Scenario
4. THE System SHALL remove the "boros" (lavish) Scenario from generation
5. THE System SHALL update the AI prompt to request only 2 scenarios
6. THE System SHALL update the response schema to expect 2 scenarios
7. THE System SHALL display both scenarios in the Planner view with equal visual prominence
8. WHEN the Planner generates scenarios, THE System SHALL complete processing within 10 seconds

### Requirement 7: Trend Chart Data Parser

**User Story:** As a developer, I want to parse transaction data into time-series format, so that the Trend_Chart can render spending patterns correctly.

#### Acceptance Criteria

1. THE System SHALL provide a parser function that accepts an array of Transactions and a view mode parameter
2. WHEN the view mode is "daily", THE Parser SHALL group transactions by date (YYYY-MM-DD)
3. WHEN the view mode is "weekly", THE Parser SHALL group transactions by ISO week number
4. WHEN the view mode is "monthly", THE Parser SHALL group transactions by month (YYYY-MM)
5. THE Parser SHALL sum nominal values for each time period
6. THE Parser SHALL return an array of objects with time_period and total_spending properties
7. THE Parser SHALL sort the output array in chronological order
8. WHEN no transactions exist for a time period, THE Parser SHALL include that period with zero spending

### Requirement 8: Budget Progress Calculator

**User Story:** As a developer, I want to calculate budget progress accurately, so that visual indicators and notifications trigger at the correct thresholds.

#### Acceptance Criteria

1. THE System SHALL provide a calculator function that accepts Category, month, and budget limit parameters
2. THE Calculator SHALL query the transaksi table for all pengeluaran transactions matching the Category and month
3. THE Calculator SHALL query the transaksi_split table for all splits matching the Category and month
4. THE Calculator SHALL sum amounts from both tables to get total spending
5. THE Calculator SHALL compute percentage as: (total_spending / budget_limit) × 100
6. THE Calculator SHALL return an object with total_spending, budget_limit, percentage, and status properties
7. WHEN percentage is less than 80, THE Calculator SHALL set status to "safe"
8. WHEN percentage is between 80 and 99, THE Calculator SHALL set status to "warning"
9. WHEN percentage is 100 or greater, THE Calculator SHALL set status to "exceeded"

### Requirement 9: Notification Trigger Service

**User Story:** As a developer, I want a service that checks spending conditions and triggers notifications, so that Users receive timely alerts.

#### Acceptance Criteria

1. THE System SHALL provide a notification service that runs budget threshold checks
2. THE Service SHALL check all active budgets whenever a new Transaction is added
3. WHEN a budget crosses the 80%, 90%, or 100% threshold, THE Service SHALL trigger a Notification
4. THE Service SHALL track which thresholds have already triggered to avoid duplicate notifications
5. THE Service SHALL check daily spending against the daily average at the end of each day
6. THE Service SHALL check for missing transactions at 8:00 PM daily
7. THE Service SHALL respect User notification preferences from localStorage
8. WHEN notifications are disabled, THE Service SHALL not send any Notification
9. THE Service SHALL use the browser Notification API to display notifications
10. THE Notification SHALL include a title, body message, and icon

### Requirement 10: Split Transaction Form Component

**User Story:** As a developer, I want a reusable form component for creating split transactions, so that the UI is consistent and maintainable.

#### Acceptance Criteria

1. THE System SHALL provide a Split_Form component that accepts transaction details as props
2. THE Split_Form SHALL display input fields for each split: Category, nominal, and catatan
3. THE Split_Form SHALL provide an "Add Split" button to add additional splits
4. THE Split_Form SHALL provide a "Remove" button for each split except the first
5. THE Split_Form SHALL display the remaining amount to be allocated
6. WHEN the sum of splits equals the total nominal, THE Split_Form SHALL enable the submit button
7. WHEN the sum of splits does not equal the total nominal, THE Split_Form SHALL display an error message
8. THE Split_Form SHALL validate that each split nominal is greater than zero
9. THE Split_Form SHALL validate that each split has a selected Category
10. THE Split_Form SHALL return an array of split objects when submitted

### Requirement 11: Database Migration Scripts

**User Story:** As a developer, I want migration scripts for the new database tables, so that the schema can be updated safely in all environments.

#### Acceptance Criteria

1. THE System SHALL provide a migration script to create the budget_kategori table
2. THE System SHALL provide a migration script to create the transaksi_split table
3. THE budget_kategori table SHALL have columns: id (SERIAL PRIMARY KEY), kategori (VARCHAR(50) NOT NULL), limit (BIGINT NOT NULL CHECK > 0), bulan (VARCHAR(7) NOT NULL), created_at (TIMESTAMPTZ DEFAULT NOW())
4. THE transaksi_split table SHALL have columns: id (SERIAL PRIMARY KEY), transaksi_id (INTEGER NOT NULL REFERENCES transaksi(id) ON DELETE CASCADE), kategori (VARCHAR(50) NOT NULL), nominal (BIGINT NOT NULL CHECK > 0), catatan (TEXT DEFAULT ''), created_at (TIMESTAMPTZ DEFAULT NOW())
5. THE migration scripts SHALL be idempotent using CREATE TABLE IF NOT EXISTS
6. THE migration scripts SHALL create appropriate indexes for query performance
7. THE System SHALL create an index on budget_kategori(kategori, bulan)
8. THE System SHALL create an index on transaksi_split(transaksi_id)
9. THE System SHALL create an index on transaksi_split(kategori, created_at)

### Requirement 12: Trend Chart Rendering Library

**User Story:** As a developer, I want to use a charting library that supports responsive line charts, so that the Trend_Chart displays correctly on all devices.

#### Acceptance Criteria

1. THE System SHALL use a charting library that supports line charts with time-series data
2. THE Chart SHALL render using SVG or Canvas for performance
3. THE Chart SHALL support responsive width and height
4. THE Chart SHALL display axis labels, grid lines, and data point markers
5. THE Chart SHALL support hover interactions to show exact values
6. THE Chart SHALL use the application's color scheme for consistency
7. THE Chart SHALL render within 1 second for datasets up to 365 data points
8. THE Chart SHALL support animation when data updates
